# XGBoost Studio 开发规范 v1.0

> 本规范适用于 XGBoost Studio 项目全周期开发，目标是确保 AI 辅助一键生成的代码可直接构建出符合验收标准的可执行应用。

---

## 一、分支策略（Git Flow 精简版）

### 1.1 分支结构

```
main          ← 永远保持可构建、可运行状态，只接受 PR 合并
dev           ← 日常集成分支，功能完成后合并到此，通过测试后合并 main
feat/模块名   ← 功能开发分支，一个模块一个分支
fix/描述      ← Bug 修复分支
```

### 1.2 分支命名规范

| 类型 | 命名格式 | 示例 |
|------|---------|------|
| 功能开发 | `feat/module-{编号}-{模块英文名}` | `feat/module-1-data-import` |
| Bug 修复 | `fix/{简短描述}` | `fix/excel-multisheet-crash` |
| 文档更新 | `docs/{描述}` | `docs/update-readme` |
| 构建/配置 | `chore/{描述}` | `chore/setup-electron-builder` |

### 1.3 分支生命周期

```
创建 feat 分支 → 开发 → 本地验证 → PR 合并到 dev → dev 通过集成测试 → 合并到 main → 打 tag
```

**规则**：
- 禁止直接向 `main` 提交代码
- 每个功能模块独立分支，开发完成后立即合并，不堆积
- `main` 每次合并后必须能执行 `npm run build` + `pyinstaller` 无报错

## 二、提交规范（Conventional Commits）

### 2.1 提交格式

```
<type>(<scope>): <subject>

[body - 可选]
[footer - 可选]
```

### 2.2 type 类型

| type | 含义 | 触发场景 |
|------|------|---------|
| `feat` | 新功能 | 完成一个验收标准中的功能点 |
| `fix` | Bug 修复 | 修复功能缺陷或验收失败项 |
| `test` | 测试 | 添加或修改测试代码 |
| `refactor` | 重构 | 不改变功能的代码优化 |
| `perf` | 性能优化 | 优化响应速度、内存占用等 |
| `docs` | 文档 | 更新 README、注释等 |
| `chore` | 构建/配置 | 依赖更新、构建脚本、CI 配置 |
| `style` | 代码风格 | 格式化，不影响逻辑 |

### 2.3 scope（影响范围）

对应验收标准文档中的模块编号：

| scope | 对应模块 |
|-------|---------|
| `data-import` | 模块1：数据导入与预览 |
| `feature-analysis` | 模块2：特征分析 |
| `feature-engineering` | 模块3：特征工程与选择 |
| `param-config` | 模块4：参数配置 |
| `model-training` | 模块5：模型训练 |
| `model-eval` | 模块6：模型评估 |
| `model-tuning` | 模块7：模型调优 |
| `model-mgmt` | 模块8：模型管理 |
| `report` | 模块9：分析报告 |
| `prediction` | 模块10：交互式预测 |
| `global` | 全局/基础设施 |
| `server` | Python 服务端 |
| `electron` | Electron 主进程 |
| `ui` | 通用 UI 组件 |

### 2.4 提交示例

```bash
# 完成一个完整功能点
feat(data-import): 实现 CSV/Excel 拖拽上传与多Sheet选择 [AC-1-01, AC-1-02]

# 修复验收失败
fix(model-eval): 修复分类任务混淆矩阵百分比计算错误 [AC-6-03]

# 性能优化
perf(data-import): 大文件(100k行)虚拟滚动渲染，渲染时间从8s优化至1.2s [AC-P-03]

# 基础设施
chore(global): 初始化 Electron+React+FastAPI 项目脚手架
```

### 2.5 何时提交

| 场景 | 是否提交 |
|------|---------|
| 完成一个验收标准用例（AC-x-xx）的全部功能 | ✅ 立即提交 |
| 完成一个模块的所有验收用例 | ✅ 提交 + 合并到 dev |
| 修复一个 Bug | ✅ 立即提交 |
| 代码未完成但下班/切换任务 | ✅ 用 `WIP:` 前缀提交到 feat 分支 |
| 代码无法编译运行 | ❌ 不提交到 dev/main |
| 仅改了缩进/空格 | ❌ 不需要单独提交，合并到下次 |

  └── feat/module-4-param-config    → AC-4-01 ~ AC-4-04

## 五、代码质量规范

### 5.1 前端（TypeScript / React）

| 规则 | 说明 |
|------|------|
| 类型安全 | 禁止使用 `any`，API 响应必须有 Pydantic 对应的 TS 类型 |
| 组件规范 | 函数式组件 + Hooks，单文件不超过 300 行，超过则拆分 |
| 状态管理 | 页面内状态用 `useState`，跨页面状态用 Zustand store |
| API 调用 | 统一封装在 `src/api/` 目录，页面组件不直接调用 `fetch` |
| 错误处理 | 所有 API 调用必须有 try/catch，错误用 Ant Design `message.error` 展示 |
| 图表 | 统一使用 ECharts，配置项抽离为独立函数，不内联在 JSX 中 |
| 加载态 | 所有异步操作必须有 loading 状态，使用 Ant Design `Spin` 或 `Skeleton` |

### 5.2 后端（Python / FastAPI）

| 规则 | 说明 |
|------|------|
| 类型注解 | 所有函数参数和返回值必须有类型注解 |
| 请求/响应 | 必须使用 Pydantic BaseModel 定义，放在 `schemas/` 目录 |
| 路由层 | 只做参数校验和调用 service，不写业务逻辑 |
| 业务层 | 纯 Python 函数，不依赖 FastAPI，方便单独测试 |
| 异常处理 | 使用 FastAPI `HTTPException`，返回标准错误格式 `{"detail": "错误信息"}` |
| 文件操作 | 上传文件保存到 `{项目目录}/data/`，路径存 SQLite，不存绝对路径 |
| SSE 流 | 训练/调优进度使用 `StreamingResponse` + `yield`，格式：`data: {json}\n\n` |
| 日志 | 使用 Python `logging` 模块，级别 INFO 以上写文件，DEBUG 仅控制台 |

### 5.3 通用规范

| 规则 | 说明 |
|------|------|
| 编码 | 所有文件 UTF-8 编码 |
| 语言 | 注释、提交信息、日志均用中文，代码变量/函数名用英文 |
| 密钥/路径 | 禁止硬编码绝对路径，用相对路径或环境变量 |
| 端口 | Python 服务默认端口 `18899`，仅监听 `127.0.0.1` |
| 依赖 | 新增 npm 包需说明用途；新增 Python 包必须更新 `requirements.txt` |

## 六、验收对齐规范（最重要）

### 6.1 开发与验收标准绑定

每个功能点开发前，必须先阅读对应的验收标准（`验收标准文档.md`）：

```
开发流程：
  1. 读 AC-x-xx 验收标准 → 明确"预期结果"清单
  2. 后端先行：实现 API + 单元测试，返回值格式匹配验收期望
  3. 前端对接：UI 展示符合"预期结果"中每一个 checkbox
  4. 手动验收：逐条勾选验收标准 checkbox
  5. 全部通过 → 提交 + 合并
```

### 6.2 验收标准追踪表

每个提交的 subject 中括号注明关联的验收用例编号：

```bash
feat(data-import): 实现缺失值处理面板 [AC-1-05]
feat(data-import): 实现异常值检测与高亮 [AC-1-06, AC-1-07]
```

### 6.3 性能基准必须在阶段 5 达标

验收标准中的性能指标（`AC-P` 系列）是硬性通过条件：

| 指标 | 目标值 | 测试方法 |
|------|-------|---------|
| 100k 行文件导入 | < 10 秒 | DS-05 计时 |
| 模型训练（DS-01） | < 30 秒 | 实测计时 |
| 应用冷启动 | < 5 秒 | 从双击到可操作 |
| PDF 报告导出 | < 30 秒 | 实测计时 |
| 内存占用（静止） | < 500 MB | 任务管理器 |

## 七、构建与打包规范

### 7.1 构建流程（Windows）

```powershell
# 步骤 1：构建 Python 服务为单一可执行文件
cd server
pyinstaller build.spec
# 输出：server/dist/xgboost-server.exe

# 步骤 2：将服务端可执行文件复制到 Electron 资源目录
cp server/dist/xgboost-server.exe client/resources/

# 步骤 3：构建 Electron 安装包
cd client
npm run build
# 输出：dist/XGBoost Studio Setup x.x.x.exe
```

### 7.2 一键构建脚本

所有步骤封装在 `scripts/build-all.ps1`，执行后产出：
- `dist/XGBoost Studio Setup {version}.exe` — Windows 安装包

### 7.3 版本号规范

遵循 [语义化版本](https://semver.org/lang/zh-CN/)：

```
{major}.{minor}.{patch}
  major：不兼容的重大变更（如重构架构）
  minor：新增功能模块（每完成一个阶段 +1）
  patch：Bug 修复

当前开发版本起点：0.1.0（阶段0完成后）
阶段1完成：0.2.0
阶段2完成：0.3.0
阶段3完成：0.4.0
阶段4完成：0.5.0
阶段5验收通过：1.0.0
```

### 7.4 Release 流程

```bash
# 阶段完成后打 tag
git tag -a v0.2.0 -m "feat: 完成数据链路模块1-3，通过AC-1~AC-3共26条验收标准"
git push origin v0.2.0
```

## 十、依赖管理与镜像源配置

### 10.1 Python 依赖管理工具：uv（推荐）

本项目 **统一使用 uv** 管理 Python 环境，不使用 conda base 环境直接开发（conda base 存在 scipy DLL 冲突问题）。

| 工具 | 用途 | 说明 |
|------|------|------|
| **uv** | 开发/CI 环境依赖管理 | 速度极快，生成 `uv.lock` 精确锁定版本 |
| **PyInstaller** | 生产打包 | 将 uv 虚拟环境打包为单一 `.exe` |
| conda | 不用于本项目 | 仅作为宿主机 Python 发行版存在 |

### 10.2 配置国内镜像源（一次性，全局生效）

#### uv 镜像源（清华大学）

```powershell
# 配置文件路径：%APPDATA%\uv\uv.toml
# 已在本机配置完成，新成员克隆后执行以下命令：
$uvConfig = "$env:APPDATA\uv\uv.toml"
New-Item -ItemType Directory -Force -Path (Split-Path $uvConfig)
@"
[[index]]
url = "https://mirrors.tuna.tsinghua.edu.cn/pypi/web/simple"
default = true
"@ | Set-Content -Encoding UTF8 $uvConfig
```

可选国内镜像（任选其一）：

| 镜像 | 地址 |
|------|------|
| 清华大学（推荐） | `https://mirrors.tuna.tsinghua.edu.cn/pypi/web/simple` |
| 阿里云 | `https://mirrors.aliyun.com/pypi/simple` |
| 腾讯云 | `https://mirrors.cloud.tencent.com/pypi/simple` |
| 华为云 | `https://mirrors.huaweicloud.com/repository/pypi/simple` |

#### npm 镜像源（淘宝）

```powershell
# 设置全局 npm 镜像，之后所有 npm install 自动走国内源
npm config set registry https://registry.npmmirror.com

# 验证
npm config get registry
```

### 10.3 server 环境操作命令

```powershell
# 首次初始化（已完成，无需重复）
cd server
uv init --python 3.12

# 安装/更新所有依赖（根据 pyproject.toml）
uv sync

# 新增一个包（自动更新 pyproject.toml 和 uv.lock）
uv add <包名>

# 删除一个包
uv remove <包名>

# 在隔离环境中运行服务
uv run python main.py

# 导出 requirements.txt（供参考，实际用 uv.lock）
uv pip compile pyproject.toml -o requirements.txt
```

### 10.4 git 追踪策略

| 文件 | 是否提交 | 说明 |
|------|---------|------|
| `server/pyproject.toml` | ✅ 提交 | 依赖声明文件 |
| `server/uv.lock` | ✅ 提交 | 精确版本锁定，确保构建可复现 |
| `server/.python-version` | ✅ 提交 | Python 版本声明 |
| `server/.venv/` | ❌ 不提交 | 虚拟环境目录，加入 .gitignore |
