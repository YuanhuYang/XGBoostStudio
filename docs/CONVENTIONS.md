# `docs/` 目录规范

> **说明**：凡是存入 `docs/` 和 `.cursor/skills/` 的文件，必须使用 **UTF-8 编码，不允许 GBK/GB2312**
> **最后更新**：2026-04-06 | **编码**：UTF-8 无 BOM

## 1. 目录结构

| 目录 | 用途 | Cursor Skill `@` 引用 |
|------|------------|----------------------------|
| **product/** | **需求/规格/验收**文档 | - |
| **guides/** | **工程指南** | - |
| **iterations/** | 按迭代 ID 组织的过程文档 | - |
| **evidence/** | 跨迭代质量/验证数据 | - |

## 2. 文件命名

1. **文件名全部使用英文或拼音命名**，避免 `需求文档.md`、`验收标准文档.md` 等中文命名

2. **会议/日志文件用日期命名**，如 `2026-04-01-meeting.md` 等，原因如下：
   - 文件名含中文在跨系统环境中经常因编码问题乱码
   - 日期命名便于按时间排序查找
   - 避免与迭代 ID 文件夹名称冲突

3. **迭代子目录下推荐四种标准文件**：
   - `章程.md`：迭代目标和验收标准
   - `设计.md`：架构和技术方案
   - `执行记录.md`：实施/测试/验收记录
   - `证据.md`：质量证据或验收截图

4. **产品文档放在**统一 `product/RELEASE_CHECKLIST.md` 等

## 3. 与 Agent Skill 配合

在文档中引用 Skill 时，推荐在表格中**明确指向操作**：

```markdown
| 操作名称 | 引用 @xs-role-xxx
```

## 4. 编码

- **必须 UTF-8 无 BOM 编码**
- **换行**：用 LF 而非 CRLF，避免 git 提交时产生二进制差异
- git 仓库 `.gitattributes` 示例：
  ```
  *.md text eol=lf
  ```

## 5. 链接

内部链接统一格式：

- 规范：`[CONVENTIONS.md](CONVENTIONS.md)`
- 产品：`[需求文档.md](product/需求文档.md)`
- 避免使用绝对路径或 URL 链接

## 6. 常见编码问题排查

如果文档上传 GitHub 后出现乱码：

1. 确认文件是否以 GBK 保存，GitHub 默认显示 UTF-8，会乱码
2. 使用 Python 脚本修复：`python scripts/fix-encoding.py docs`
3. 修复后重新验证文件是 UTF-8 编码
