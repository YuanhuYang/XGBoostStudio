# 迭代规划 · 模型权威性与专业性（项目经理）

> **目标**：让数据分析专家用户难以从 **可复现性、统计严谨性、方法论透明度、调优可解释性、报告诚实度** 五方面挑出「不专业」的毛病；与 **G1 信任链**（[`章程.md`](../iterations/G1/章程.md)）、**G2-R1 报告叙事** 衔接，不重复造轮子。  
> **原则**：**先可证、再好看**；所有「权威」声明必须能落到 **版本钉、随机种子、算式引用、pytest/记录** 之一。

### 当前阶段（项目经理）

- **G2-Auth-1**：已关闭（见上）。  
- **G2-Auth-2 · 首版已落地并含 GAP-G2-AUTH-002 闭环**：`evaluation_protocol`（含时间序列划分元数据）、基线 train-fit、`POST /api/training/kfold`、训练期可选 K 折（AC-6-03）与评估页展示、特征工程页时间序划分；pytest + 矩阵 + [`执行记录.md`](../iterations/G2-Auth-2/执行记录.md)。  
- **G2-Auth-3 · 已落地**：Optuna 调优 `diagnostics`（trial 序列、失败计数、`search_space_documentation`）、禁止静默失败；API/结果持久化；pytest + [`执行记录.md`](../iterations/G2-Auth-3-4/执行记录.md)；章程 [`章程.md`](../iterations/G2-Auth-3/章程.md)。  
- **G2-Auth-4 · 已落地**：PDF `methodology` 章节、条件化业务建议、与叙事同源的因果/多重比较免责；[`report_methodology.py`](../server/services/report_methodology.py)；pytest + 同上执行记录；章程 [`章程.md`](../iterations/G2-Auth-4/章程.md)。  
- **下一步排期**：持续 G1 全量验收 / G2-R1b 演进等按 [`迭代计划.md`](迭代计划.md) 与缺口表。

## 与专家挑刺的对应关系

| 专家可能质疑的点 | 产品应答方向（迭代切片） |
|------------------|-------------------------|
| 同样数据两次结果不一致 | **全链路可复现**（种子、划分、训练、调优、SHAP） |
| 指标怎么算的、是否数据泄漏 | **评估协议透明**（train/val/test 边界、仅训练集做 EDA 叙事已部分落地） |
| 默认参数/推荐值拍脑袋 | **引用层级**：官方默认 → 规则推荐（样本量分层）→ 文献/Optuna 搜索空间 **显式文档化** |
| 调优是否收敛、是否过搜 | **Optuna 诊断**（试验曲线、最优 trial 对比默认基线） |
| 报告夸大、因果误读 | **叙事与建议的限定语**（已有免责；加强「方法」小节与局限性） |
| 与 sklearn/xgboost 最佳实践差太远 | **权威基准集回归**（扩展 `test_authority_*`、OpenML 小任务） |

## 建议拆分为 4 个可商用迭代（可并行轨道）

### 迭代 A · G2-Auth-1「可复现与运行档案」（P0）

**交付**：任意一次训练/调优可导出 **运行档案**（建议 JSON）：`xgboost/sklearn/python` 版本、`dataset_id`/`split_id`、**全随机种子**（划分、训练、`numpy`/`random`）、`params` 最终合并结果、`metrics`、可选 `git_commit`（若 CI 注入）。

**研发落点**：[`training_service`](../server/services/training_service.py)、[`tuning_service`](../server/services/tuning_service.py)、`Model`/`TuningTask` 元数据字段；API `GET /api/models/{id}/provenance`（名称可议）。

**验收**：冻结 AC + pytest（档案字段齐全、同种子重跑指标一致或 documented 漂移）；**五领域**：系统与契约、建模与评估（种子策略）、质量门禁（快照/区间）。

**周期建议**：1～2 周（小步：先训练种子，再调优种子）。

---

### 迭代 B · G2-Auth-2「评估与验证协议」（P0）

**交付**：  
- **分层 K 折 / 时间序列划分** 产品与文档；hold-out 方差说明 + **可选训练期 K 折**（见 AC-6-03 与 [`迭代计划.md`](迭代计划.md) GAP-G2-AUTH-002 **已完成**）。  
- **基线模型**（Dummy / 多数类）与当前模型 **同框报告**（已有部分对比能力则 **强化为默认章节**）。  
- **指标定义脚注**（PDF/帮助）：Accuracy/AUC/RMSE 公式与适用前提。

**验收**：AC 对齐 [`验收标准文档.md`](验收标准文档.md) 模块 6/3 相关条；pytest 用固定划分断言「优于 dummy」或 **登记豁免**（不平衡极端集）。

**周期建议**：2～3 周（与数据划分 UI 改动量相关）。

---

### 迭代 C · G2-Auth-3「调优可信度」（P1）

**交付**：Optuna **study 摘要**（最佳 trial、n_trials、收敛图）；**搜索空间** 与 XGBoost 文档/团队规范 **对照表**；禁止静默失败（trial 失败计入日志并展示计数）。

**验收**：端到端 pytest（小 `n_trials`）；手工：专家审查搜索空间合理性（**建模与评估** 领域）。

**周期建议**：2 周。

---

### 迭代 D · G2-Auth-4「报告与建议的专业表述」（P1，可与 A 交错）

**交付**：PDF 增加 **「方法与指标定义」** 固定页；**业务建议** 与指标 **条件化**（例：仅当 AUC>0.7 时建议上线试验）；与 [`dataset_narrative_service`](../server/services/dataset_narrative_service.py) 的 **多重比较/因果** 话术统一由模板库维护（单源字符串）。

**验收**：文案走 **产品体验** 双视角样例（信息架构 + 可学性）；pytest 做关键词/结构存在性（已有 PDF 锚点模式可参考）。

**周期建议**：1～2 周。

## 跨迭代不变量（项目经理盯盘）

1. **契约优先**：任何新指标/推荐规则进 **OpenAPI + 验收-AC证据矩阵 + pytest**。  
2. **升级依赖**：`pyproject.toml` 变更 → 重跑 **权威集回归**（乳腺癌、Titanic 等）并更新区间或快照说明。  
3. **共识溢出**：只认 [`迭代计划.md`](迭代计划.md) **GAP-G1-CONSENSUS-001** 与缺口表，避免口头加 scope。  
4. **与 G1 全量验收关系**：本路线图 **不替代** [`功能需求与验收状态.md`](功能需求与验收状态.md) §二/§四；商用发布仍走原门禁。
