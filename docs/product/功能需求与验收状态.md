# 功能需求与验收状态（六轮交付对齐）

> **用途**：把 [`迭代计划.md`](迭代计划.md) 中已实现能力沉淀为可追踪矩阵，并与 [`验收标准文档.md`](验收标准文档.md) 模块/AC 对应，支撑 **全量验收** 与发布门禁。  
> **原则**：**发布前**须本表「全量验收状态」对当期发布范围全部为 **通过**（或经项目经理书面豁免并记风险）；不得以「代码已合并」替代「验收已执行」。  
> **全量**：须对 [`验收标准文档.md`](验收标准文档.md) 当期适用 **全部 AC** 执行并留痕；历史 **抽样**（如 F3）**不能**替代全量结论。**模块顺序、批次与日程由项目经理排期**，QA（**质量门禁** 行与 QA Owner 为同一人时填写）按排期执行。  
> **术语**：签核采用 **五领域**（与 Cursor Subagent 一致），定义与历史「六列」对照见 [`项目管理术语-五领域.md`](项目管理术语-五领域.md)。

**对外表述门禁（硬规则）**：仅当 **第二节** 表格中、**当期发布口径** 所覆盖各行的「全量验收状态」**全部为「通过」**，且 **第四节五领域签核表** 已填写结论（「通过 / 附条件 / 不通过」及日期；或有经批准的书面豁免清单并登记风险）后，方可在 **对外材料**（官网、商务与合同附件、发版说明、对客演示材料等）中使用「**迭代计划范围已验收通过**」或语义等价的表述。否则只能描述为「功能已交付 / 开发完成 / 部分验收通过」等，并指向本表与执行记录。

## 五、下一工作包（建议迭代 **G1：全量验收冲刺**）

1. **冻结范围**：仅执行与记录验收，**默认不改产品功能**（发现 P0 缺陷单开缺陷迭代）。  
2. **顺序与日程**：**须全量覆盖** [`验收标准文档.md`](验收标准文档.md) 当期适用 AC；**推荐参考顺序** 为模块 0 → 1 → 5 → 6 → 9 → 10 → 其余模块，**具体排期由项目经理下达**（可与推荐顺序不一致，以书面排期为准）。  
3. **产出物**：在对应 `docs/iterations/<迭代ID>/` 更新 **`执行记录.md`**（正文可含日期与环境）；更新本表第二节「全量验收状态」列；更新 [`RELEASE_CHECKLIST.md`](RELEASE_CHECKLIST.md)。  
4. **完成定义**：[`迭代计划.md`](迭代计划.md) 第六节「验证标准（商业验收）」所列条目 + 所选发布口径下全部相关 AC 有勾选/记录与签核。  
5. **QA / 测试把控**：**质量门禁（QA）行签核人兼任 QA Owner**；按 [`迭代计划.md`](迭代计划.md) **「G1 · QA / 测试把控」** 与 **「G1 · 往下执行」** 执行：CI 门禁、A/B 档 AC、证据与 §二对齐；阻塞项分级上报。  
6. **未实现需求**：验收中若发现与 [`需求文档.md`](需求文档.md) 或迭代实现声明不符，**不得**将对应行标为「通过」；由 QA 在 [`迭代计划.md`](迭代计划.md) **验收缺口登记** 表 **加一行**，**计划迭代** 标 **G1 / G1.1 / G2**（见同文档 **「需求未实现时的排期」**）。  
7. **双线（产品 vs 质量）**：**G1 文档与全量 AC 不阻塞产品迭代**——研发可持续做 **G2+** 优化（[`迭代计划.md`](迭代计划.md) **G2+** 小节）；**商用对外发布** 仍须单独满足本表 §二+§四 与 [`RELEASE_CHECKLIST.md`](RELEASE_CHECKLIST.md)。  
8. **商用发布一页核对**：发版窗口前由项目经理 / QA 按 [`RELEASE_CHECKLIST.md`](RELEASE_CHECKLIST.md) **§0 商用发布质量保障** 勾选，避免遗漏门禁。
