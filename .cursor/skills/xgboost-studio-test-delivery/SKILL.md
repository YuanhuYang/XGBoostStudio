# 测试交付（分析 → 设计 → 自动化 → 全量 → 修复）

## 何时使用

- 用户要求 **全量测试**、**测试分析/设计**、**自动化脚本**、**测完有问题就修**、**避免假完成**。

通常在 **`xgboost-studio-rd-implementation`**（或等价已实现范围）之后使用；测试分析须 **覆盖本轮已实现模块**。

在 **Harness 多迭代** 模式下：以 **《迭代章程》** 为准——**阶段 4 全量执行** = 章程规定的 **本迭代必跑命令子集** + **强制回归清单**（不必每迭代都等同「仓库历史上全部测试」，但必须在文档中显式列出并实际执行）；最终产品级全量可在 **里程碑迭代** 单独定义。

## 与 `xgboost-studio-testing` 的关系

- `xgboost-studio-testing`：日常测点与分层策略。
- **本 Skill**：强调 **完成定义（DoD）**、**证据**、**全量矩阵** 与 **缺陷闭环**。两者同时适用时以本 Skill 的 DoD 为准。

## 阶段 1 — 测试分析（基于代码与文档）

1. 枚举 **功能模块**：后端 API 路由组、前端页面/流程、Electron 特有行为。
2. 建立 **需求-实现映射**：每个用户可见能力对应到文件/路由（找不到则记为缺口）。
3. 标出 **高风险区**：上传/解析、训练任务、文件系统、并发、端口与路径。

输出：**《测试范围说明》**（表格：模块 | 风险 | 已有覆盖 | 缺口）。

## 阶段 2 — 测试设计

- 为每个模块写 **可验收测点**（Given/When/Then 或 输入/期望/错误码）。
- 分层标注：**单元 / API 集成 / E2E（最小集）**。
- 明确 **测试数据**：使用 `server/tests/fixtures/`，禁止依赖本机隐私路径。

输出：**《测例设计表》**（与阶段 1 表格可合并或双向链接）。

## 阶段 3 — 自动化实现

- 优先 **pytest + httpx/TestClient**（后端）；前端 **typecheck + 后续 Vitest/Playwright**（按仓库现状迭代）。
- 每个用例必须有 **失败时能定位的断言**（忌只断言 200）。
- 服务依赖：用 **fixture 起停** 或 **TestClient**；若端口硬编码（如 `18899`），在测试中 **集中配置** 或 mock。

输出：**可重复执行的测试命令** 写入 `.cursor/AGENTS.md`（或本 Skill 维护的「命令区块」由该文件引用）。

## 阶段 4 — 全量执行

1. 运行文档中的 **全量命令**（后端 `pytest`、客户端 `npm run typecheck`、验收脚本若适用则单独标明「需已起服务」）。
2. 记录 **完整输出摘要**：通过数、失败栈、 flaky 提示。
3. **禁止**在未实际执行的情况下写「已通过」。

## 阶段 5 — 缺陷闭环

- 对每个失败：**复现步骤 → 根因 → 修复产品或修复测试（说明理由）→ 重跑证明**。
- 若研发声称已修复：必须 **同命令重跑 + 针对该缺陷的回归用例**。

## 完成定义（DoD）— 防「假完成」

在宣称「本阶段测试完成」前必须满足：

- [ ] 《测试范围说明》中 **本迭代 P0 范围**（或章程冻结范围）均有自动化或经批准的豁免理由（豁免需记录风险）。
- [ ] **章程/本文档约定的命令集合** 在本环境 **实际执行过**，并粘贴或通过日志文件证明关键结果（Harness 迭代可为子集 + 回归，但必须写明且不得瞒报）。
- [ ] 无 **已知未关联 issue 的失败用例**；跳过的用例列出清单。
- [ ] 契约变更已同步（OpenAPI/类型/前端调用处至少一处可追溯）。
- [ ] **商用/可交付**：满足本迭代《迭代章程》中的 **交付定义**（若未使用 Harness，则默认等同「可合并且无 P0 缺陷」）。

## 并行与拆解

- **分析/设计** 可与 **实现** 分 PR，但 **全量执行 + DoD** 必须在同一轮收尾。
- 多 Agent 时：一人维护 **测例设计表** 为单一事实来源，避免重复或矛盾用例。
