# 研发实现（评审 → 代码）

## 何时使用

- 已完成 **《优化建议指南》** 或等价需求列表，需要 **落地功能模块** 再进入全量测试。
- 用户提示词中的 **阶段 B（研发实现）**。

## 输入

- **《优化建议指南》** 中的 **P0/P1**，或 **Harness** 下本迭代的 **《迭代章程》**（路线图中的 **范围冻结清单**）；二者取其一为权威范围，**超出清单即蔓延，默认不做**。
- **开放问题** 未关闭的项不得强行用代码臆断，应标注 TODO 或停手问人。

## 执行原则

1. **范围冻结（本轮）**：列出本迭代要交付的模块/接口/页面，避免顺手重构无关代码。
2. **契约先行**：先定或对齐 API（路径、请求/响应、错误码）再写前后端；变更写在注释或简短 dev note（可选 `.cursor/AGENTS.md` 旁注），供测试阶段引用。
3. **小步提交**：每完成一个可编译、可局部验证的单元再继续；优先 `server` 可启动、`client` `npm run typecheck` 通过。
4. **自测最小集**：实现后本地跑 **与改动相关** 的命令（如单模块脚本、手工调关键 API）；**不等同于** 阶段 C 的全量测试与 DoD。
5. **与指南对齐**：每条实现对应指南中的一条建议或明确需求，便于追溯（可在 PR 描述或清单中勾连）。

## 完成定义（本阶段 DoD，与测试阶段区分）

- [ ] P0 相关项已落地或可证明刻意延期（写明原因与后续 issue）。
- [ ] 前后端主路径可跑通（或 mock 边界已标明）。
- [ ] `typecheck` / 服务端启动无新增阻塞性错误（以仓库现状为准）。

全量自动化测试与 **测试阶段 DoD** 由 **`xgboost-studio-test-delivery`** 负责。
