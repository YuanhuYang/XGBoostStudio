# 迭代优化方向 → Subagent / Skill 映射

> **目的**：向 PM 给出方向时，快速选对 **子代理、Skill 与文档**，避免「只有一个 /pm 不知道下一步 @ 谁」。  
> **原则**：多数方向是 **组合使用**（例如研发 + 质量）；**无**与业务领域一一对应的「第六个固定 Subagent」——**运维/流程/文档** 用下表 **归口** 到现有能力。

---

## 1. 能否覆盖？

| 你的意图（示例） | 是否已有归口 | 说明 |
|------------------|--------------|------|
| 产品研发（功能/算法/UI/API） | ✅ | 主线 **A→B→C** + `/implementation` |
| 质量专项（测试、AC、缺陷、CI 绿） | ✅ | `/quality-gate` + `/test-delivery` + 测试类 Skill |
| 用户体验优化 | ✅ | 侧重 **`/product-experience`**，可并行其他域 |
| 项目管理流程优化 | ✅ | **`/pm`** + Harness/PM Skill；流程 **成文** 找 **`/doc-steward`** |
| 需求规划与梳理 | ✅ | **`/pm`** + 路线图/章程；细化用 **`multi-agent-review`**（压缩） |
| 运维专项（部署、打包、环境、监控、脚本） | ✅（组合） | **无单独「运维」Subagent**；归口 **`/systems-contract`** + [`docs/guides/部署说明.md`](../docs/guides/部署说明.md) + [`.cursor/rules/ci-pr-babysit.mdc`](rules/ci-pr-babysit.mdc) + 可选 **`xgboost-studio-iteration-intents`** 下「运维」行 |
| 文档/规范/目录治理 | ✅ | **`/doc-steward`** + `xgboost-studio-doc-steward` |
| 版本与发版叙事 | ✅ | `xgboost-studio-release-versioning` + [`版本与发布.md`](../docs/product/版本与发布.md) |

## 3. 运维专项：归口说明（重要）

本仓库 **不** 单独提供 `/ops` Subagent，避免与 **系统与契约** 重复。运维类迭代请：

1. **`/systems-contract`**：打包、进程、端口、Electron 与后端边界、可观测性挂钩的契约。  
2. **文档**：[`docs/guides/部署说明.md`](../docs/guides/部署说明.md)、[`docs/product/RELEASE_CHECKLIST.md`](../docs/product/RELEASE_CHECKLIST.md) 安装包相关节。  
3. **CI/CD**：改 `.github` 时启用规则 **`ci-pr-babysit`**；与 Skill **`xgboost-studio-testing`** 中的 CI 说明一致。  
4. 若任务以 **脚本/工作流** 为主、少改产品代码：仍可用 **`/implementation`** 小步提交，但审视以 **`systems-contract`** 为准。

## 4. 复合方向（常见）

| 组合 | 建议顺序 |
|------|----------|
| 研发 + 质量 | 范围冻结后：`/implementation` 与 `/test-delivery` **同迭代内**串或分会话；合并前 **C 阶段 DoD**。 |
| 体验 + 产品 | 先 `/product-experience`（A）再 `/implementation`（B）。 |
| 需求梳理 + 文档治理 | `/pm` 定范围 → `/doc-steward` 落 `docs/` 结构。 |
| 流程优化 + 版本发版 | `/pm` + `xgboost-studio-release-versioning` + `RELEASE_CHECKLIST`。 |

---

## 5. 与五领域审视的关系

「产品研发」类默认仍走 **五领域 A 阶段**，但 **按迭代冻结范围裁剪**：无关域由 PM 标 **跳过**（见 `multi-agent-review`、`.cursor/README` § Subagents）。**质量 / 体验 / 运维** 专项可 **优先** 启用对应行（quality-gate / product-experience / systems-contract），不必五次全开。
